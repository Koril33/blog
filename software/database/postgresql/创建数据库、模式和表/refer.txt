# 🧱 2. PostgreSQL 的逻辑结构（核心概念）

👉 这一章是全文最重要的

---

## 2.1 分层模型（必须画图）

```text
Cluster
 └── Database
      └── Schema
           └── Table
```

讲清：

* Database = 逻辑隔离单元
* Schema = 命名空间（namespace）
* Table = 数据存储

---

## 2.2 类比（非常重要）

👉 用文件系统类比：

| PostgreSQL | 类比      |
| ---------- | ------- |
| Database   | 硬盘 / 分区 |
| Schema     | 文件夹     |
| Table      | 文件      |

👉 这一段能让新手“瞬间理解”

## 3.2 指定 owner


## 3.3 常见参数（点到为止）

# 📦 4. Schema（很多人最容易懵的地方）

---

## 4.1 什么是 Schema？

👉 定义：

> Schema 是数据库中的命名空间

## 4.2 为什么需要 Schema？

重点讲这三个场景：

### ✅ 1. 业务隔离


## 4.3 创建 Schema


## 4.4 schema + owner


## 4.5 search_path（⚠️ 必讲）

👉 这是新手最大坑之一

```sql

```

解释：

* 为什么可以直接 `SELECT * FROM table`
* 实际是 `schema.table`

# 🧪 7. 一个完整示例（强烈建议）

👉 用一个真实场景串起来：

```text
创建一个应用：
- database: app_db
- schema: app
- 表: users / orders
- 用户: app_user
```

---

示例流程：

```sql
CREATE DATABASE app_db;

\c app_db

CREATE SCHEMA app;

CREATE TABLE app.users (...);

CREATE USER app_user;

GRANT USAGE ON SCHEMA app TO app_user;
GRANT SELECT, INSERT ON ALL TABLES IN SCHEMA app TO app_user;
```

👉 这一节是“从0到1实战”

# ⚠️ 8. 常见误区（非常加分）

建议列出来：

👉 后果：

* 混乱
* 权限难控

### ❌ 误区 2：把 schema 当 database 用

👉 解释区别：

* schema ≠ database

---

### ❌ 误区 3：忽略 search_path

👉 导致：

* 表查错
* 安全问题

---

### ❌ 误区 4：owner 混乱

👉 后果：

* 无法删除对象
* 权限难管理

# 🧱 一、先建立一个核心认知（很重要）

在 PostgreSQL 里：

> 👉 **几乎所有对象都可以 ALTER，而不是重建**

这和很多新手的直觉不同（他们会想 DROP + CREATE）

---

# 🏢 二、ALTER DATABASE（数据库级别）

---

## 2.1 修改数据库名称

```sql
ALTER DATABASE mydb RENAME TO mydb_new;
```

⚠️ 注意：

* 不能连接到该数据库时执行
* 需要切换到其他 DB（如 `postgres`）

---

## 2.2 修改 owner

```sql
ALTER DATABASE mydb OWNER TO new_owner;
```

👉 这个在团队交接 / 权限整理时很常见

---

## 2.3 修改配置参数

```sql
ALTER DATABASE mydb SET work_mem = '64MB';
```

👉 用于：

* 性能调优
* 特定 DB 定制参数

---

## ❗强调限制

👉 **不能修改：**

* encoding
* locale

👉 这些只能在 `CREATE DATABASE` 时指定

---

# 📦 三、ALTER SCHEMA

---

## 3.1 重命名 schema

```sql
ALTER SCHEMA old_schema RENAME TO new_schema;
```

---

## 3.2 修改 owner

```sql
ALTER SCHEMA my_schema OWNER TO new_user;
```

---

## 3.3 ⚠️ 影响

👉 schema 改名会影响：

* 所有 `schema.table` 引用
* 应用代码（如果写死）

---

# 🗄️ 四、ALTER TABLE（最重要）

👉 这是 ALTER 的核心部分，建议重点写

---

## 4.1 添加列

```sql
ALTER TABLE users ADD COLUMN email TEXT;
```

---

## 4.2 删除列

```sql
ALTER TABLE users DROP COLUMN email;
```

⚠️ 注意：

* 数据直接丢失（不可恢复）

---

## 4.3 修改列类型

```sql
ALTER TABLE users ALTER COLUMN age TYPE BIGINT;
```

👉 常见场景：

* int → bigint
* varchar → text

⚠️ 风险：

* 可能锁表
* 可能失败（类型不兼容）

---

## 4.4 修改默认值

```sql
ALTER TABLE users ALTER COLUMN created_at SET DEFAULT now();
```

---

## 4.5 删除默认值

```sql
ALTER TABLE users ALTER COLUMN created_at DROP DEFAULT;
```

---

## 4.6 设置 NOT NULL

```sql
ALTER TABLE users ALTER COLUMN name SET NOT NULL;
```

⚠️ 前提：

* 现有数据不能有 NULL

---

## 4.7 添加约束

```sql
ALTER TABLE users ADD CONSTRAINT users_email_unique UNIQUE (email);
```

---

## 4.8 删除约束

```sql
ALTER TABLE users DROP CONSTRAINT users_email_unique;
```

---

## 4.9 重命名表

```sql
ALTER TABLE users RENAME TO app_users;
```

---

## 4.10 修改表 owner

```sql
ALTER TABLE users OWNER TO app_user;
```

---

# ⚠️ 五、生产环境必须讲的内容（很加分）

这一部分建议单独一小节：

---

## 5.1 锁（Lock）

👉 很关键：

```text
ALTER TABLE 很多操作会锁表
```

影响：

* 读写阻塞
* 线上请求卡住

---

## 5.2 大表风险

例如：

```sql
ALTER TABLE big_table ADD COLUMN ...
```

👉 可能：

* 扫全表
* 持续几分钟甚至更久

---

## 5.3 安全策略（建议写出来）

### ✅ 分步修改

```text
1. 加 nullable 列
2. 回填数据
3. 再加 NOT NULL
```

---

### ✅ 避免高峰操作

---

### ✅ 使用事务（部分场景）

# 🧪 六、结合一个真实演进案例（强烈推荐）

👉 这一节会让文章“非常实用”

---

## 场景：给 users 表加 email 字段

---

### ❌ 错误做法

```sql
ALTER TABLE users ADD COLUMN email TEXT NOT NULL;
```

👉 会失败（已有数据）

---

### ✅ 正确做法

```sql
ALTER TABLE users ADD COLUMN email TEXT;

UPDATE users SET email = 'default@example.com';

ALTER TABLE users ALTER COLUMN email SET NOT NULL;
```

---

👉 这类案例比纯语法更有价值
